# 申请付款

## 做什么

将当前游戏账户余额发起付款申请，写入游戏付款管理，并将相关结算记录改为「已申请」。

## 显示条件

仅当该行**账户余额 > 0** 时显示。

## 点击校验（红 Toast，不弹窗）

| 顺序 | 条件 | 提示 |
|------|------|------|
| 1 | 归属厂商银行信息未填全 | 未填写银行信息 |
| 2 | 游戏未填写预付分成款（0 合法） | {游戏名称}未补充预付分成款信息 |
| 3 | 游戏付款管理存在该游戏未付款记录 | 存在一笔未付款的记录 |

校验通过后弹出二次确认。

## 确认成功后的逻辑

在点击「确认申请」的**这一时刻**，系统按当时信息生成付款记录和结算函，并**固定保存**。之后即使游戏资料、预付分成、付款设置等发生变化，**本笔付款与结算函内容不变**（与是否已标记付款无关）。

### 当时固定保存的信息

| 分类 | 内容 |
|------|------|
| 游戏 | 游戏编号、游戏名称、**支付币种**（申请付款那一刻的快照值） |
| 厂商 | 厂商编号、厂商名称、银行收款信息 |
| 预付分成 | 预付分成款、**剩余未抵扣预付分成款**（与游戏收入列表当时一致） |
| 付费设置 | **付款币种**、分成付款公司、付款账号 |
| 时间与金额 | 申请时间、**待付款金额**（= 申请时账户余额） |
| 结算明细 | 本次纳入的已结算收入/退款明细 |
| 结算函 | 整函内容（明细、合计、汇率、抵扣行、初始实际付款金额等） |

### 联动效果

| 页面/模块 | 发生什么 |
|-----------|----------|
| 游戏付款管理 | 主列表**最上方**新增一条「未付款」记录 |
| 内部/外部/退款结算 | 本次纳入的明细，申请付款状态改为「已申请」 |
| 游戏收入管理 | 该游戏账户余额清零（已申请明细不再计入余额） |
| 结算函 | 随时打开均显示申请时保存的内容，不随后台数据变 |
| 标记付款 | 分成付款公司、收款信息、实付初始值均按申请时保存的信息展示（实付填写值写入付款记录，不反写结算函） |

标记已付款后，结算函**全部内容不变**。
