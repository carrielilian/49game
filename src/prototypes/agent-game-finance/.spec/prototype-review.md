## 总体点评

代理游戏台账原型已形成较完整的业务闭环：厂商/游戏主数据 → 合同与结算公式 → 内外部结算分叉（外部导入弹窗结算、内部拉取后列表结算）→ **游戏**收入核算与申请付款 → **游戏**付款管理（标记付款/结算函/请款凭证/详细信息）→ 收入汇总统计。主流程入口清晰，关键校验均有 Toast 反馈，适合作为需求评审与验收演示载体。

**2026-07-24 复审结论**：厂商收入、厂商付款管理已完整删除；`vendor-income` / `payment-list` 不再注册路由，旧深链回到默认厂商管理页。游戏收入、游戏付款及其结算函流程继续保留。

独立评估：**pass（degraded）** — 基于源码与资源文档对照，无 P0/P1 阻塞项；剩余为演示级简化与产品待确认项。

## 业务逻辑连贯性

- **账户余额**（`balance.ts` / `deriveGameBalances`）：未申请结算收入 − 未申请退款；**不扣**预付分成款。
- **游戏名称**：「游戏ID / 游戏名称」取 `onlineName`；「合同游戏名称」取 `name`。
- **支持渠道**：仅内部渠道 6 项（无 H5游戏）；入口在游戏管理操作列。
- **合同管理**：合同编号/金额/合作内容/条件已付；**支付币种**在合同管理中维护（`Contract.currency`）；未设置币种时金额输入前缀为空；合同金额未填不显示 0.00。
- **收入页金额**：结算口径列固定 ￥；预付未填列表显示 `-`。
- **收入汇总**：结算付款金额 = 游戏付款已付款 `actualAmount` 按付款月累加。
