## 总体点评

代理游戏台账原型已形成较完整的业务闭环：厂商/游戏主数据 → 合同与结算公式 → 内外部结算分叉（外部导入弹窗结算、内部拉取后列表结算）→ 厂商收入核算与申请付款 → 付款管理（标记付款/结算函/详细信息）→ 收入汇总统计。主流程入口清晰，关键校验（表单必填、申请付款三级拦截、内部结算公式校验、外部导入前置校验）均有 Toast 反馈，适合作为需求评审与验收演示载体。

当前最影响业务成立的问题集中在**付款与结算的数据关联**和**结算函口径**：从付款管理打开的结算函按厂商汇总全部已结算记录计算总额，与付款行的 `pendingAmount` 可能不一致；申请付款时批量将厂商全部「未申请」结算记录改为「已申请」，但 `PaymentRequest` 未记录关联结算 ID，后续无法追溯「这笔付款对应哪些结算」。此外，资源文档（conversation-memory、ui-spec）尚未同步本轮实现变更（平台更名、统计页侧栏隐藏、详细信息字段拆分），评审与后续开发若以文档为准会产生误判。

原型在空数据、校验失败、按钮禁用等状态上有基本覆盖；作为 mock 原型，网络/权限/并发等系统性异常未实现属预期范围。独立评估为 **degraded**（无子代理并行，业务基线与源码证据在同一轮次综合完成）。

## P0-P3 优先级问题

### P1 - 结算函金额口径与付款申请记录不一致

- 证据：`PaymentListPage.tsx` 打开结算函时传入 `amount={current.pendingAmount}`；`SettlementLetterDrawer.tsx` 实际汇总该厂商全部 `settled` 结算记录的收入/退款行计算「总计②-④」，仅在无结算记录时才回退使用传入 `amount`。
- 影响：财务人员在付款管理查看结算函时，文档底部支付金额可能与列表「待付款金额」不符，削弱结算函作为付款依据的可信度，验收时易被判为业务逻辑错误。
- 修复方向：明确结算函应对齐哪一业务口径——（a）仅包含本次申请付款涉及的结算记录，或（b）厂商当期全部已结算明细；在 `PaymentRequest` 与结算函组件间建立关联或过滤条件，并同步更新 `ui-spec.md` 口径说明。

### P1 - 申请付款与结算记录无关联，状态批量变更不可追溯

- 证据：`data/types.ts` 中 `PaymentRequest` 无 `settlementIds` 等关联字段；`store.tsx` 的 `applyPayment` 创建付款记录后，将该厂商所有 `settled && paymentApplyStatus === '未申请'` 的结算行一次性改为「已申请」，无明细绑定。
- 影响：无法回答「这笔 P00x 付款对应哪些 S00x 结算记录」；若后续支持部分申请、撤销或复核，现有模型无法扩展；测试验收难以核对金额来源。
- 修复方向：在申请付款时写入关联结算 ID 列表（或快照金额明细）；结算函、详细信息抽屉可据此过滤展示；补充 mock 验收样例与字段说明。

### P2 - 标记付款允许实际付款金额脱离待付款金额且无上限校验

- 证据：`PaymentListPage.tsx` 标记付款抽屉中「待付款金额」可编辑，校验仅要求 `>0` 且最多两位小数；`store.tsx` 的 `markPaid` 将输入值写入 `actualAmount`，不校验是否等于或不超过 `pendingAmount`。
- 影响：列表「待付款金额」与「实际付款金额」可能长期不一致，且列表 `pendingAmount` 在标记已付款后不会更新，造成同一付款记录两套金额语义并存，增加财务对账摩擦。
- 修复方向：区分字段语义——标记付款编辑项应明确为「实际付款金额」；增加 `actualAmount <= pendingAmount`（或业务允许的偏差规则）校验；或在标记已付款后同步更新/锁定 `pendingAmount`。

### P2 - 详细信息抽屉未覆盖列表中的付款状态与时间字段

- 证据：付款列表含「付款状态」「申请付款时间」「付款时间」三列（`PaymentListPage.tsx`）；详细信息抽屉仅展示厂商信息、待付款/实际付款金额、付款银行、收款信息、附件与备注，缺少状态与时间只读字段。
- 影响：用户从「详细信息」入口无法完成付款记录的全貌核对，需返回列表对照，业务状态表达不完整。
- 修复方向：在详细信息抽屉补充只读字段：付款状态、申请付款时间、付款时间（格式与列表一致，精确至秒）；已付款记录展示 `payTime`，待付款展示 `-`。

### P2 - 资源文档与当前实现存在多处偏差（待确认）

- 证据：
  - 代码侧栏与面包屑已使用「代理游戏台账」（`Sidebar.tsx`、`index.tsx`），`conversation-memory.md` / `ui-spec.md` 仍称「代理游戏财务」/「代理游戏财务平台」。
  - 本轮用户要求隐藏厂商/渠道/游戏收入统计，`index.tsx` 侧栏仅保留「收入汇总统计」，但两文档仍写侧栏 4 个统计菜单；对应路由 `#page=stats-vendor` 等仍可通过 hash 直达。
  - `ui-spec.md` 详细信息「待付款金额」仍写 `actualAmount ?? pendingAmount` 单字段，实现已拆分为「待付款金额」+「实际付款金额」两个只读字段（本轮用户指令）。
- 影响：后续对话、评审或开发若仅读资源文档，会对产品命名、统计范围、付款详情字段产生错误预期。
- 修复方向：以当前实现与用户最新指令为准，回写 `conversation-memory.md` 与 `ui-spec.md`；并确认隐藏统计页是「菜单不可见但保留路由」还是「彻底下线」。

## 业务逻辑连贯性

**内外部结算路径**自洽且与文档定稿一致：外部禁止主列表未结算数据，导入确认写 `settled: true`；内部/退款依赖财务中心拉取（mock）后列表结算，按钮状态存于 `store.internalSettlementButtons`，切换菜单不重置、刷新重置。

**厂商收入余额**（`utils/balance.ts`）与列表展示一致：账户余额 = 未申请结算收入 − 未申请退款 − 预付分成款；申请付款成功后相关结算行改「已申请」，余额归零逻辑成立。但 `payments` 参数在 `deriveBalances` 中未参与计算，当前仅依赖 `paymentApplyStatus` 翻转，与 P1 关联缺失问题同源。

**游戏名称双字段**（`onlineName` / `name`）在列表、抽屉、搜索中执行一致；合作/运营状态枚举未回退旧值。

**会话态标志**：`externalSettlementButtons.settleCompleted` 在外部导入弹窗点击【结算】后置 true，用于厂商收入申请付款确认文案；该标志与是否实际导入数据无绑定，可能导致「未导入但已结算」的误提示消除，属于演示级简化。

**内部【结算】范围**仅处理当前筛选后的未结算行（`InternalSettlementPage.tsx`），与 ui-spec 一致；若筛选条件收窄，屏幕外未结算行不会被处理——业务上需知悉此为「当前列表批量结算」而非「全量结算」。
